# BabyLog 产品设计准则

本文档总结 BabyLog 的产品设计经验、UX 规范和开发流程约定。编码规范见 `docs/02-Architecture/overview.md`，视觉规范见 `docs/05-UI-Style-Guide/index.md`。

---

## 一、产品设计原则

### 1. 场景驱动，不是功能堆砌

BabyLog 的用户是**正在带宝宝的父母**——可能单手抱着孩子、可能凌晨 3 点半睡半醒、可能在嘈杂的儿科候诊室。设计每个功能时先问：

- 用户在什么场景下用这个功能？（喂奶中？宝宝刚睡着？）
- 用户的手是自由的吗？（可能只有一只手）
- 用户的精力充足吗？（可能严重缺觉）
- 用户操作完需要多快回到现实？（宝宝随时可能哭）

### 2. 每一步都有反馈

用户操作后**必须**立即知道结果。这不是"锦上添花"，对于睡眠不足的父母来说，没有反馈 = 不确定 = 焦虑。

- 成功操作 → Toast 提示
- 破坏性操作 → 确认弹窗
- 网络错误 → 错误 Toast
- 加载中 → 按钮禁用 + loading 状态

### 3. 最少步骤完成任务

父母的时间以秒计算。每多一步操作，就多一次宝宝可能打断的机会。

- 只有一个宝宝时，自动选中，跳过选择步骤
- 记录完成后智能跳转，不让用户迷失在导航里
- 高频操作（记录喂奶、换尿布）放在最容易触达的位置
- 计时器一键开始，结束后自动填充数据

### 4. 空状态是引导，不是死胡同

新用户或数据为空时，不能只显示一句话。每个空状态都是**引导用户完成下一步**的机会。

- 没有宝宝 → "去添加宝宝"按钮（可点击跳转，不只是文字）
- 没有记录 → "开始记录"引导
- 时间线为空 → 友好提示 + 引导到快速记录

### 5. 为分享而设计

每个记录功能在设计时就要想：这条记录如果被爷爷奶奶看到，信息够不够？

- 事件卡片显示关键信息（类型、时间、详情、照片）
- 里程碑用特殊样式突出，值得被"翻阅"
- 照片是记忆的核心载体，鼓励但不强制

## 二、UX 规范

### 反馈机制

| 操作类型 | 反馈方式 | 示例 |
|---------|---------|------|
| 创建成功 | Toast "success" | 添加宝宝、记录事件 |
| 删除/退出 | Alert 确认 → Toast | 删除宝宝、退出登录 |
| 表单校验失败 | Alert 提示 | "请输入宝宝姓名" |
| 网络错误 | Toast "error" | "网络连接失败" |
| 复制操作 | Toast "已复制" | 复制 Token |

### 导航流转

- 快速记录完成 → 回到来源页（首页/时间线）+ Toast
- 添加宝宝完成 → 留在设置页 + Toast + 列表刷新
- 注册完成 → 显示 Token 页（强制用户看到 Token 和安全提醒）→ 点击进入首页
- 退出登录 → 确认弹窗 → 回到 Welcome 页

### 设置页分区

设置页按功能分组，每组用 section 标题 + 卡片容器：

1. **应用设置**：FaceID/TouchID 开关、语言切换、深色模式（预留）
2. **宝宝管理**：宝宝列表（卡片化，带头像）+ 添加宝宝入口
3. **账号与安全**：个人称呼/头像、Token 状态、退出登录

新增设置项时按此分组归入，不要在页面底部随意追加。

### 时间线交互

- 默认按时间倒序（最新在上）
- 筛选芯片：全部 / 各宝宝名 / 日常 / 里程碑
- 事件卡片长按 → 操作菜单（编辑/删除）
- 里程碑卡片用特殊视觉样式，区别于日常记录
- 照片缩略图网格，点击全屏浏览

### 国际化（i18n）

- 所有文案使用 `t('key')`，禁止硬编码
- 新增功能时**先加翻译 key 再写 UI**
- key 按页面分组：`common.*`、`home.*`、`quickAdd.*`、`settings.*`、`timeline.*`
- 设置页提供三选项：中文 / English / 跟随系统

## 三、开发流程约定

### UX 审计流程

每完成一个阶段的功能开发，执行以下审计流程：

```
研发完成 → 测试更新 E2E 并截图 → 产品审查截图 → 发现问题回研发修复
```

1. **测试**：修改 Playwright E2E，在每个关键步骤插入 `page.screenshot()`
2. **测试**：输出截图到 `apps/mobile/e2e/screenshots/`，生成 HTML 审计报告（`index.html`）
3. **产品**：浏览报告，标注 UX 问题
4. **研发**：修复问题
5. **测试**：截图对比"优化前 vs 优化后"

### 研发与测试分工

- `[研发]`：功能实现（前端 + 后端 + 类型定义）
- `[测试]`：E2E 测试编写 + 截图审计 + Bug 修复验证
- 测试任务标注 `_(等 x.x~x.x 完成)_` 表示阻塞依赖

### 新功能上线检查清单

每个功能实现完毕，确认以下项目：

1. **i18n**：zh.json 和 en.json 都有对应 key
2. **testID**：关键元素在 `testids.ts` 注册并在 JSX 中设置
3. **Toast 反馈**：成功/失败都有 toast
4. **确认弹窗**：破坏性操作有 Alert 确认
5. **空状态**：数据为空时有友好引导
6. **theme**：颜色和间距引用 theme token，无硬编码值
7. **类型定义**：`packages/shared-types` 同步更新
8. **E2E**：跑通 Playwright 测试，无回归

## 四、产品路线图哲学

### 每一期是下一期的基础

```
v0.2 丰富记录 + 照片 → v0.3 才有内容值得分享给亲友
v0.3 家庭成员 + 角色 → v1.0 才有用户身份支撑社区
```

不跳步。如果某个功能依赖另一个功能还没做好的基础设施，优先做基础设施。

### 本地优先，服务端跟上

对于需要持久化的用户偏好（语言、称呼、头像），先用 AsyncStorage 做本地版快速上线，下一期服务端 ready 后再迁移。

- v0.2：用户称呼和头像存本地
- v0.3：`family_members` 表就绪后迁移到服务端

### 功能复用优于新建

能复用已有数据模型就不新建表：

- 里程碑复用 `events` 表（`event_type = 'milestone'`），统一时间线展示
- 照片存在 `events.payload.photoUrls`，不单独建 `photos` 表

### 向前兼容

新功能的数据结构要考虑未来扩展：

- `events.payload` 是 JSON，方便未来加字段而不改 schema
- 用户资料本地版的数据格式要和 `family_members` 表兼容
- i18n key 的命名留足层级空间
